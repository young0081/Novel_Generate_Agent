# 原著对齐功能实施路线图

## 概述

本文档规划"原著对齐增强"功能的实施步骤，解决同人创作中"Agent 经常偏离原著"的问题。

**目标**：让 AI 生成的内容"像原作续写"，不 OOC。

---

## 当前状态

✅ **已完成**：
- Story State 系统（解决"忘设定"问题）
- 状态自动注入机制
- 基础的约束管理

🔄 **待实施**：
- 原著锚点数据结构
- 原著对齐 Prompt 注入
- 一致性审校器

---

## Phase 1: 数据结构扩展（2-3 天）

### 目标
扩展现有 `StoryState` 数据结构，支持原著锚点建模。

### 任务清单

#### 1.1 扩展 CharacterState（1 天）
- [ ] 在 `na-story/src/state.rs` 中添加：
  - `CharacterCanon` 结构
  - `SpeechStyle` 结构
  - `MotivationPattern` 结构
  - `RelationshipPattern` 结构
  - `EmotionStyle` 枚举
  - `CanonSceneRef` 结构
- [ ] 为 `CharacterState` 添加 `canon: Option<CharacterCanon>` 字段
- [ ] 实现序列化/反序列化
- [ ] 编写单元测试（至少 5 个）

#### 1.2 扩展 WorldState（0.5 天）
- [ ] 添加 `WorldCanon` 结构
- [ ] 添加 `CoreSetting`, `PowerSystem` 结构
- [ ] 为 `WorldState` 添加 `canon: Option<WorldCanon>` 字段
- [ ] 单元测试

#### 1.3 添加 CreativeFreedom（0.5 天）
- [ ] 在 `StoryState` 中添加 `creative_freedom` 字段
- [ ] 实现 `AllowedChange`, `LockedAspect` 结构
- [ ] 实现 `FreedomLevel` 枚举
- [ ] 单元测试

#### 1.4 示例数据（1 天）
- [ ] 创建完整的示例 JSON（福尔摩斯 + 华生）
- [ ] 创建示例 JSON（哈利波特角色）
- [ ] 验证序列化/反序列化正确

**验收标准**：
- ✅ 所有新结构可正确序列化为 JSON
- ✅ 从 JSON 加载无错误
- ✅ 至少 10 个单元测试通过
- ✅ 示例数据完整且合理

## Phase 2: Prompt 增强（1-2 天）

### 目标
实现原著对齐 Prompt 的渲染和注入。

### 任务清单

#### 2.1 Prompt 渲染函数（1 天）
- [ ] 在 `na-story/src/prompts.rs` 中实现：
  - `render_canon_alignment_prompt()`
  - `render_creative_freedom_prompt()`
  - 渲染角色性格核心
  - 渲染说话风格
  - 渲染行为边界
  - 渲染动机逻辑
- [ ] 单元测试：验证 Prompt 格式正确

#### 2.2 注入逻辑修改（0.5 天）
- [ ] 修改 `desktop-tauri/src-tauri/src/lib.rs` 中的 `run_goal_live`
- [ ] 添加 `has_canon_data()` 检查
- [ ] 在状态注入后添加原著对齐注入
- [ ] 确保注入顺序：writer.md → 状态 → 原著对齐 → 用户目标

#### 2.3 集成测试（0.5 天）
- [ ] 创建测试场景（含 canon 数据的 story_state.json）
- [ ] 运行 `run_goal_live`，检查会话消息
- [ ] 验证原著对齐 Prompt 正确出现在会话中
- [ ] 手动测试：生成一章，检查角色是否更贴原著

**验收标准**：
- ✅ Prompt 渲染清晰易读
- ✅ 注入位置正确（在用户消息前）
- ✅ 测试：有 canon 时注入，无 canon 时跳过
- ✅ 桌面端编译零错误

## Phase 3: 基础审校器（MVP）（2-3 天）

### 目标
实现最基础的原著对齐检查（仅 OOC 和行为边界）。

### 任务清单

#### 3.1 数据结构（0.5 天）
- [ ] 在 `na-story/src/guard.rs` 中添加：
  - `CanonAlignmentReport` 结构
  - `CanonIssue` 结构
  - `CanonIssueType` 枚举

#### 3.2 审校 Prompt 模板（1 天）
- [ ] 实现 OOC 检查 Prompt
- [ ] 实现行为边界检查 Prompt
- [ ] 测试：用 mock 返回验证 Prompt 格式

#### 3.3 基础检查逻辑（1 天）
- [ ] 实现 `check_character_ooc()`
- [ ] 实现 `check_behavior_boundaries()`
- [ ] 调用模型 API，解析返回的 JSON
- [ ] 汇总为 `CanonAlignmentReport`

#### 3.4 集成到生成流程（0.5 天）
- [ ] 修改 `run_goal_live`，在生成后调用审校
- [ ] 返回审校报告给前端
- [ ] 前端显示：overall_score + issues 列表

**验收标准**：
- ✅ 能检测出明显的 OOC（如福尔摩斯突然主动表达感情）
- ✅ 能检测出行为边界违反（如角色做了"绝不会做的事"）
- ✅ 返回清晰的问题描述和修改建议
- ✅ 桌面端可显示审校结果

---

## Phase 4: 完整审校器（进阶）（3-5 天）

### 目标
实现全面的原著对齐检查。

### 任务清单

#### 4.1 说话风格检查（1 天）
- [ ] 实现说话风格审校 Prompt
- [ ] 实现 `check_speech_style()`
- [ ] 单独测试：能识别语气/用词/句式不符

#### 4.2 关系张力检查（1 天）
- [ ] 实现关系审校 Prompt
- [ ] 实现 `check_relationship_fidelity()`
- [ ] 测试：能识别关系失真

#### 4.3 世界观一致性检查（1 天）
- [ ] 实现世界观审校 Prompt
- [ ] 实现 `check_world_canon()`
- [ ] 测试：能识别设定违反

#### 4.4 评分体系（1 天）
- [ ] 实现加权评分算法
- [ ] 实现等级划分（0.9-1.0 完美 / 0.0-0.5 严重OOC）
- [ ] 根据评分自动决策（通过/建议修改/强烈建议重写）

#### 4.5 综合测试（1 天）
- [ ] 准备 10 个测试章节（5 个贴原著 + 5 个 OOC）
- [ ] 运行完整审校流程
- [ ] 验证评分准确性（贴原著的 ≥0.8，OOC 的 ≤0.6）

**验收标准**：
- ✅ 能全方位检查原著对齐性
- ✅ 评分客观且有区分度
- ✅ 测试准确率 ≥80%

## Phase 5: UI 集成（3-4 天）

### 目标
为原著锚点管理和审校结果提供友好界面。

### 任务清单

#### 5.1 状态管理界面扩展（2 天）
- [ ] 在"状态管理"屏添加"原著锚点"标签
- [ ] 角色卡片显示：
  - 性格核心编辑
  - 说话风格编辑（语气/用词/示例台词）
  - 行为边界列表（增删改）
  - 动机逻辑模式（增删改）
- [ ] 世界观部分：
  - 核心设定编辑
  - 权力体系编辑
- [ ] 保存/加载功能

#### 5.2 审校报告界面（1 天）
- [ ] 创建 `CanonAlignmentReportView` 组件
- [ ] 显示 overall_score（进度条 + 等级）
- [ ] 列出所有 issues（按严重性排序）
- [ ] 每个 issue 显示：
  - 问题类型（OOC/说话风格/行为边界等）
  - 具体描述
  - 问题片段
  - 修改建议
- [ ] 支持"一键定位到问题位置"

#### 5.3 创作流程集成（1 天）
- [ ] 生成完成后自动显示审校报告
- [ ] 根据评分显示不同提示：
  - ≥0.8：绿色，"优秀，高度贴原著"
  - 0.6-0.8：黄色，"建议检查以下问题"
  - <0.6：红色，"严重偏离，强烈建议修改"
- [ ] 提供"重新生成"和"手动修改"选项

**验收标准**：
- ✅ 用户可轻松编辑原著锚点
- ✅ 审校报告清晰直观
- ✅ 问题定位准确

## Phase 6: 自动修正（可选，3-5 天）

### 目标
根据审校报告自动修正 OOC 内容。

### 任务清单

#### 6.1 修正 Prompt（1 天）
- [ ] 设计"原著对齐修正" Prompt
- [ ] 输入：原文 + 问题列表 + 原著锚点
- [ ] 输出：修正后的文本

#### 6.2 修正逻辑（2 天）
- [ ] 实现逐段修正（避免一次性处理太长）
- [ ] 实现 `auto_fix_canon_issues()`
- [ ] 修正后重新审校验证

#### 6.3 UI 集成（1 天）
- [ ] 添加"自动修正"按钮
- [ ] 显示修正前后对比
- [ ] 用户确认后应用修正

**验收标准**：
- ✅ 能自动修正 70% 以上的问题
- ✅ 修正后不引入新问题
- ✅ 用户可选择接受或拒绝修正

## 风险与应对

### 风险 1：审校准确率不足
**应对**：
- 多轮迭代 Prompt
- 收集真实案例优化
- 考虑引入人工标注数据

### 风险 2：性能问题（审校太慢）
**应对**：
- 并行检查多个维度
- 缓存模型调用结果
- 提供"快速模式"（只检查关键项）

### 风险 3：用户不愿意填写原著锚点
**应对**：
- 提供 AI 自动提取功能
- 提供常见 IP 的预设模板
- 简化输入（只要求必填项）
