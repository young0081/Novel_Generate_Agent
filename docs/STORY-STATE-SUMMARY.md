# Story State 系统实施总结

## 🎯 目标达成

**问题陈述**：
> "越到后面的剧情，问题越大" — 用户反馈
> 
> 核心症状：角色 OOC、前后逻辑不一致、时间线混乱、剧情约束失效、伏笔/信息状态管理混乱
> 
> 根本原因：系统没有把设定和约束稳定地带到后续生成里

**解决方案**：
✅ 完全实现。创建了完整的 Story State 系统，通过结构化状态管理 + 自动注入机制，确保 AI 在每次生成时都能看到完整且一致的剧情状态。

## 🏗️ 系统架构

### 数据流

```
创作流程:
用户点击"创作" 
  → 前端调用 invoke('run_goal_live', {goal, title})
  → 后端检查 workspace/story_state.json
  → 如存在: 加载状态 → 准备上下文 → 渲染 Prompt
  → 注入到会话为系统消息
  → AI 看到: [writer.md] → [状态同步] → [用户目标]
  → 生成严格遵守设定的内容
```

### 核心类型

```rust
StoryState {
    meta: StoryMeta,                    // 标题、类型、当前章节
    world: WorldState,                  // 世界规则
    characters: HashMap<CharacterState>, // 角色状态
    timeline: Timeline,                 // 时间线
    knowledge_matrix: KnowledgeMatrix,  // 谁知道什么
    foreshadows: Vec<ForeshadowTracker>, // 伏笔追踪
    hard_constraints: Vec<Constraint>,  // 硬约束（5 级）
    soft_preferences: Vec<Preference>,  // 软偏好
    current_chapter_goal: Option<...>,  // 本章目标
}
```

### 状态注入示例

**注入前**（旧系统）：
```
[System: writer.md 风格指导]
[User: 写第五章：突破筑基]
```

**注入后**（新系统）：
```
[System: writer.md 风格指导]
[System: 剧情状态同步 - 角色/约束/伏笔/事件]  ← 新增
[User: 写第五章：突破筑基]
```

## 🔧 技术亮点

### 1. 自动状态注入
不需要用户手动粘贴设定，系统在每次创作前自动注入最新状态。实现在 `run_goal_live` 中：
```rust
let state_path = workspace_root.join("story_state.json");
if state_path.exists() {
    if let Ok(mgr) = StoryStateManager::open(&state_path) {
        let chapter_num = mgr.state.meta.last_chapter.saturating_add(1);
        let ctx_pkg = mgr.prepare_context(chapter_num);
        let state_prompt = render_state_sync_prompt(&ctx_pkg);
        session.push(Message::system(state_prompt));
    }
}
```

### 2. 智能上下文过滤
- 只注入 **High 及以上** 的约束（减少 token）
- 只显示 **最近 3 个** 事件（保持简洁）
- 只列出 **未回收** 的伏笔（聚焦相关内容）
- 约束按 **优先级降序** 排列（Critical 最前）

### 3. 知识矩阵
防止角色"凭空得知"不该知道的信息：
```rust
knowledge_matrix.set_knowledge("char_001", "fact_secret", false);
// 如果 AI 让 char_001 知道 fact_secret → 检测为知识泄漏
```

### 4. 伏笔生命周期管理
```rust
pub enum ForeshadowStatus {
    Planted,   // 刚埋下
    Hinted,    // 已暗示
    Resolved,  // 已回收
    Abandoned, // 放弃
}
```

### 5. 5 级约束严重性
```rust
pub enum Severity {
    Info = 1,      // 提示性
    Low = 2,       // 可违反但不推荐
    Medium = 3,    // 应该遵守
    High = 4,      // 必须遵守
    Critical = 5,  // 绝对不可违反
}
```

## 🚀 实际使用流程（当前可用）

### 1. 创建初始状态

用户在 `workspace/` 下创建 `story_state.json`：
```json
{
  "meta": {"title": "修仙逆袭录", "genre": "修仙", "last_chapter": 0},
  "characters": {"char_001": {"name": "林惊羽", "core_traits": ["冷静", "重情义"], ...}},
  "hard_constraints": [
    {"id": "hc_001", "description": "主角绝不背叛朋友", "severity": "Critical"}
  ],
  ...
}
```

### 2. 开始创作

用户在桌面端"创作"界面输入：
```
写第一章：师傅遇害，主角立誓复仇
```

### 3. 系统自动处理

```
[后台] 检测到 story_state.json
[后台] 加载状态
[后台] 准备第 1 章上下文
[后台] 渲染状态 Prompt
[后台] 注入到会话
[AI 看到]
  System: writer.md 风格
  System: 剧情状态（角色、约束、伏笔）  ← 自动注入
  User: 写第一章...
[AI 生成] 严格遵守角色特征和约束的内容
```

### 4. 更新状态（手动/未来自动）

章节完成后，更新状态：
```rust
let mut mgr = StoryStateManager::open("workspace/story_state.json")?;
mgr.add_timeline_event(1, "师傅被害，主角立誓复仇".to_string());
mgr.plant_foreshadow("fh_001".to_string(), "师傅临终时眼神看向北方".to_string(), 1);
mgr.advance_chapter();  // last_chapter: 0 → 1
mgr.save()?;
```

### 5. 继续创作第二章

下次创作时，AI 会看到更新后的状态，包括第一章埋下的伏笔和新的时间线事件。
