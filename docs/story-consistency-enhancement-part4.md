## 十二、进阶功能设想 (未来迭代)

### 12.1 智能状态提取

**当前 MVP**: 用户手动维护 StoryState

**进阶**: Agent 自动从生成的章节中提取状态更新

```rust
// 伪代码
pub async fn auto_extract_updates(
    chapter_content: &str,
    current_state: &StoryState,
    model: &dyn ModelProvider,
) -> Result<StateUpdate> {
    // 构造提取 Prompt
    let prompt = format!(
        "从以下章节中提取状态变化:\n\n{}\n\n输出 JSON:\n{}",
        chapter_content,
        STATE_UPDATE_SCHEMA
    );
    
    // 调用模型
    let response = model.complete_structured(prompt, STATE_UPDATE_SCHEMA).await?;
    
    // 解析为 StateUpdate
    Ok(serde_json::from_value(response)?)
}
```

**提取内容**:
- 新发生的事件 → 追加到 Timeline
- 角色获知的信息 → 更新 KnowledgeMatrix
- 伏笔的暗示/回收 → 更新 ForeshadowStatus
- 角色状态变化 → 更新 CharacterState

### 12.2 剧情冲突预警

**功能**: 在生成前,分析用户的需求是否会与现有状态冲突

**示例**:
```
用户输入: "第6章,主角御剑飞到北方雪山"

系统检测到冲突:
⚠️ 硬约束违规风险
- 约束 hc_002: "筑基期以下无法御剑飞行"
- 当前状态: 主角为练气九层

建议:
1. 改为其他交通方式(徒步/租用飞舟)
2. 让主角先突破筑基境界
3. 修改约束(如果设定有误)

[继续生成 (忽略警告)] [修改需求] [调整状态]
```

### 12.3 多分支剧情 (What-If)

**场景**: 作者想尝试"如果主角选择B路线会怎样"

**实现**:
- StoryState 支持分支: `state_mainline.json`, `state_branch_dark.json`
- 从某个章节分叉,创建独立状态副本
- 两条线独立发展,互不影响

**界面**:
```
当前分支: [主线 ▼]
可用分支:
  - 主线 (当前)
  - 暗黑路线 (从第5章分出)
  - [+ 创建新分支]
```

### 12.4 协作模式

**场景**: 多位作者共同创作同一部小说

**实现**:
- StoryState 版本化 (Git-like)
- 支持合并冲突解决
- 权限管理(谁能修改硬约束)

**流程**:
```
作者A: 写第6章,更新状态
     ↓
     提交状态变更到共享仓库
     ↓
作者B: 拉取最新状态
     ↓
     写第7章 (基于A的状态)
```

### 12.5 剧情一致性评分

**功能**: 自动评估整部作品的一致性

**评分维度**:
- 角色一致性: 90/100 (主角有2次轻微 OOC)
- 时间线清晰度: 95/100
- 伏笔回收率: 75% (4个埋下,3个回收)
- 约束遵守率: 98/100 (1次违规但已修正)

**报告**:
```markdown
# 一致性评估报告

## 总分: 89/100 (良好)

### 主要问题
1. 第8章: 主角突然使用了从未提及的法术 (逻辑问题)
2. 伏笔 fh_003 未回收 (第2章埋下的"神秘玉佩",全文未提及)

### 改进建议
- 在第5章补充主角学习该法术的情节
- 决定是否回收玉佩伏笔,或标记为废弃
```

## 十三、常见问题 (FAQ)

### Q1: 这套方案会不会过于复杂,增加用户负担?

**A**: MVP 阶段的核心是"自动注入",用户无需主动操作。高级功能(状态编辑)是可选的。大部分用户只需:
1. 初次设定时填写角色/约束(5-10分钟)
2. 之后每章自动生成,系统自动维护状态
3. 只有当 Agent 出错时才需要手动纠正

### Q2: 如果 Agent 提取的状态更新有误怎么办?

**A**: 
- MVP 阶段用户手动维护,避免自动提取的错误
- 进阶阶段提供"确认更新"界面,用户审核后再应用
- 提供"回滚到上一章状态"功能

### Q3: StoryState 会不会随着章节增多而膨胀,导致加载慢?

**A**:
- 只加载"活跃"部分(当前相关角色+未回收伏笔+高优先级约束)
- Timeline 只保留最近 N 个事件,更早的归档到 MemoryStore
- 提供"清理"功能,归档已完成的伏笔和不再出场的角色

### Q4: 与现有 MemoryStore 有什么区别?

**A**:
| 维度 | StoryState | MemoryStore |
|-----|-----------|-------------|
| 用途 | 核心状态,必须存在 | 补充记忆,可召回 |
| 加载方式 | 每次直接加载 | BM25 检索按需召回 |
| 内容 | 结构化(JSON) | 非结构化文本 |
| 示例 | "主角不杀人"(约束) | "主角曾在雨夜立誓不杀人"(详细描述) |

两者互补:StoryState 保证关键信息不丢,MemoryStore 提供丰富细节。

### Q5: 能否支持多个作品?

**A**: 每个作品独立的 `story_state.json`,通过工作区路径区分:
```
workspace_project_a/
  ├── story_state.json
  ├── chapters/
  
workspace_project_b/
  ├── story_state.json
  ├── chapters/
```

### Q6: 如果中途想改设定怎么办?

**A**: 
- 小改动:直接在状态页面修改,从下一章生效
- 大改动(例如推翻世界观):
  1. 创建新分支
  2. 或使用"从第X章重写"功能

### Q7: 会增加多少 token 消耗?

**A**: 估算(以 8K 窗口为例):
- 当前系统:可用约 6000 token (扣除系统指令)
- 增强系统:约 5200 token (额外消耗 800 token 用于状态注入)
- 净减少: ~13%

但换来的是:
- ✅ 生成质量提升 → 减少重写次数 → 总 token 可能更少
- ✅ 失败率降低 → 单次就生成可用内容

## 十四、总结与核心结论

### 14.1 核心问题重述

当前 Agent 在长篇创作中"忘设定"的根本原因:
1. 设定依赖被动召回(BM25),召回失败=遗忘
2. 无结构化状态,剧情进展靠"回忆"不靠"数据"
3. 无生成前检查,约束容易被遗漏
4. 无生成后校验,错误积累到用户发现

### 14.2 解决方案精髓

**三个支柱**:
1. **结构化状态管理** (StoryState): 把"应该记住的"变成数据结构
2. **主动注入机制** (ContextPackage): 每次生成前强制加载关键约束
3. **双向校验** (ConsistencyGuard): 生成前同步,生成后检查

**实现原则**:
- 渐进式:MVP 先解决最痛的问题(硬约束遗忘)
- 自动化:尽量减少手动维护,自动提取状态变化
- 可回退:所有操作可撤销,状态有版本备份
- 透明化:让用户看到"系统记住了什么"

### 14.3 预期收益

| 维度 | 改进 | 测量指标 |
|-----|------|---------|
| 设定遗忘率 | 减少 70% | 违反设定次数 |
| 角色 OOC | 减少 80% | OOC 检测触发率 |
| 伏笔管理 | 回收率 +50% | 从 50% 提升到 80% |
| 用户纠正成本 | 减少 60% | 手动修改次数/章 |
| 整体满意度 | +1.5 分 | 5分制评分 |

### 14.4 开发时间线

**MVP (5天)**:
- Day 1: 数据结构定义
- Day 2: StoryStateManager
- Day 2.5: Prompt 注入集成
- Day 3: 硬约束优先加载
- Day 4: 基础一致性检查
- Day 5: GUI 状态编辑页面

**Beta 测试 (2周)**:
- 招募 5 位用户内测
- 收集反馈,修复 bug

**V1.0 (再 1周)**:
- 优化用户体验
- 补充文档和教程
- 正式发布

**总计: 约 1 个月可推出稳定版本**

### 14.5 最后的建议

**给开发者**:
- 优先做 MVP,验证方向再迭代
- 自动提取是难点,MVP 可先手动
- 一致性检查的 Prompt 需要反复调优

**给用户**:
- 前期多花 10 分钟设置,后期省 1 小时纠错
- 把约束分"Critical/High/Medium",不是所有都平等
- 定期查看状态页面,确认系统"记住"了什么
